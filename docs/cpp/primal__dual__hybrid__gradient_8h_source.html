35// The following table defines the interpretation of the result vectors
36// depending on the value of `solve_log.termination_reason`: (the
37// TERMINATION_REASON_ prefix is omitted for brevity):
38//
39// * OPTIMAL: the vectors satisfy the termination criteria for optimality.
40// * PRIMAL_INFEASIBLE: dual_solution and reduced_costs provide an approximate
41// certificate of primal infeasibility; see
42// https://developers.google.com/optimization/lp/pdlp_math#infeasibility_identification
43// for more information.
44// * DUAL_INFEASIBLE: `primal_solution` provides an approximate certificate of
45// dual infeasibility; see
46// https://developers.google.com/optimization/lp/pdlp_math#infeasibility_identification
47// for more information.
48// * PRIMAL_OR_DUAL_INFEASIBLE: the problem was shown to be primal and/or dual
49// infeasible but no certificate of infeasibility is available. The
50// `primal_solution` and `dual_solution` have no meaning. This status is only
51// used when presolve is enabled.
52// * TIME_LIMIT, ITERATION_LIMIT, KKT_MATRIX_PASS_LIMIT, NUMERICAL_ERROR,
53// INTERRUPTED_BY_USER: the vectors contain an iterate at the time that the
54// respective event occurred. Their values may or may not be meaningful. In
55// some cases solution quality information is available; see documentation for
56// `solve_log.solution_type`.
57// * INVALID_PROBLEM, INVALID_PARAMETER, OTHER: the solution vectors are
58// meaningless and may not have lengths consistent with the input problem.
59struct SolverResult {
60 Eigen::VectorXd primal_solution;
61 // See https://developers.google.com/optimization/lp/pdlp_math for the
62 // interpretation of `dual_solution` and `reduced_costs`.
63 Eigen::VectorXd dual_solution;
64 // NOTE: The definition of reduced_costs changed in OR-tools version 9.8.
65 // See
66 // https://developers.google.com/optimization/lp/pdlp_math#reduced_costs_dual_residuals_and_the_corrected_dual_objective
67 // for details.
68 Eigen::VectorXd reduced_costs;
69 SolveLog solve_log;
70};

99// Solves the given QP using PDLP (Primal-Dual hybrid gradient enhanced for LP).
100//
101// All operations that are repeated each iteration are executed in parallel
102// using `params.num_threads()` threads.
103//
104// The algorithm generally follows the description in
105// https://arxiv.org/pdf/2106.04756.pdf, with further enhancements for QP.
106// Notation here and in the implementation follows Chambolle and Pock, "On the
107// ergodic convergence rates of a first-order primal-dual algorithm"
108// (http://www.optimization-online.org/DB_FILE/2014/09/4532.pdf).
109// That paper doesn't explicitly use the terminology "primal-dual hybrid
110// gradient" but their Theorem 1 is analyzing PDHG. See
111// https://developers.google.com/optimization/lp/pdlp_math#saddle-point_formulation
112// for the saddle-point formulation of the QP we use that is compatible with
113// Chambolle and Pock.
114//
115// We use 0.5 ||.||^2 for both the primal and dual distance functions.
116//
117// We parameterize the primal and dual step sizes (tau and sigma in Chambolle
118// and Pock) as:
119// primal_stepsize = step_size / primal_weight
120// dual_stepsize = step_size * primal_weight
121// where step_size and primal_weight are parameters.
122// `params.linesearch_rule` determines the update rule for step_size.
123// `params.initial_primal_weight` specifies how primal_weight is initialized
124// and `params.primal_weight_update_smoothing` controls how primal_weight is
125// updated.
126//
127// If `interrupt_solve` is not nullptr, then the solver will periodically check
128// if `interrupt_solve->load()` is true, in which case the solve will terminate
129// with `TERMINATION_REASON_INTERRUPTED_BY_USER`.
130//
131// If `iteration_stats_callback` is not nullptr, then at each termination step
132// (when iteration stats are logged), `iteration_stats_callback` will also
133// be called with those iteration stats.
134//
135// Callers MUST check `solve_log.termination_reason` before using the vectors in
136// the `SolverResult`. See the comment on `SolverResult` for interpreting the
137// termination reason.
138//
139// All objective values reported by the algorithm are transformed by using
140// `QuadraticProgram::ApplyObjectiveScalingAndOffset`.
141//
142// NOTE: `qp` is intentionally passed by value, because
143// `PrimalDualHybridGradient` modifies its copy.
144SolverResult PrimalDualHybridGradient(
145 QuadraticProgram qp, const PrimalDualHybridGradientParams& params,
146 const std::atomic<bool>* interrupt_solve = nullptr,
147 std::function<void(const IterationCallbackInfo&)> iteration_stats_callback =
148 nullptr);
149
150// Like above but optionally starts with the given `initial_solution`. If no
151// `initial_solution` is given the zero vector is used. In either case
152// `initial_solution` is projected onto the primal and dual variable bounds
153// before use. Convergence should be faster if `initial_solution` is close to an
154// optimal solution. NOTE: `initial_solution` is intentionally passed by value.
155SolverResult PrimalDualHybridGradient(
156 QuadraticProgram qp, const PrimalDualHybridGradientParams& params,
157 std::optional<PrimalAndDualSolution> initial_solution,
158 const std::atomic<bool>* interrupt_solve = nullptr,
159 std::function<void(const IterationCallbackInfo&)> iteration_stats_callback =
160 nullptr);
